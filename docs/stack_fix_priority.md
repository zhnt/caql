# 🛠️ AQL栈修复优先级方案

## 🎯 问题分析

通过对比Lua和AQL的实现，发现了**根本问题**：

### 核心问题
1. **类型系统不一致**：AQL混用`StkId`和`StkIdRel`
2. **大量代码依赖错误的栈指针算术**
3. **需要系统性重构所有栈相关代码**

### 影响范围
- `src/astate.h` - 状态结构定义
- `src/ado.c` - 栈管理核心逻辑
- `src/ado.h` - 栈操作宏定义
- `src/aarray.c` - 数组操作中的栈使用
- `src/avm.c` - 虚拟机中的栈操作
- `src/aapi.c` - API中的栈操作
- 其他所有使用栈指针的文件

## 🚀 简化修复策略

考虑到这是一个**系统性重构**，我建议采用**渐进式修复**：

### 方案A：最小化修复（推荐）
1. **保持现有类型系统**：继续使用`StkId`
2. **修复upvalue转换bug**：使用正确的指针算术
3. **移除硬编码限制**：已完成
4. **优化栈增长策略**：已完成

### 方案B：完全重构（长期）
1. **统一使用StkIdRel**：需要修改所有栈相关代码
2. **完全对齐Lua实现**：工作量巨大
3. **全面测试验证**：需要大量时间

## 🎯 当前建议

**立即采用方案A**，原因：
1. **工作量可控**：只需修复核心bug
2. **风险较低**：不破坏现有架构
3. **效果明显**：能立即提升递归能力
4. **可迭代**：后续可以逐步优化

## 🔧 方案A的具体修复

### 1. 回退类型修改
暂时回退`aql_State`和`CallInfo`的类型修改，保持使用`StkId`

### 2. 修复upvalue转换
使用正确的指针算术，避免内存损坏

### 3. 优化栈检查逻辑
改进栈溢出检测和处理

这样可以在最小化风险的前提下，快速获得显著的递归能力提升。
